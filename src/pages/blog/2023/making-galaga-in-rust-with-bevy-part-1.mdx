# What‚Äôs Bevy?

[Bevy](https://bevyengine.org/) is a game engine framework used with Rust. It‚Äôs runs on [wgpu](https://wgpu.rs/), which is a Rust implementation of WebGPU, which basically means Bevy apps run not only natively ‚Äî but on the web! It‚Äôs a standard low-level game engine, expect to work with [an ECS system](https://en.wikipedia.org/wiki/Entity_component_system) to compose your 2D or 3D scene. Bevy also has a few systems in place for things like primitive shapes, handling images/audio, and other nice utilities.

In this tutorial we‚Äôll explore quite a few Bevy features, but you can also read [the Bevy ‚Äúbook‚Äù](https://bevyengine.org/learn) or browse through [the official Bevy examples](https://bevyengine.org/examples) to get a bigger and better picture. I also recommend reading [the Bevy Cheatbook](https://bevy-cheatbook.github.io/introduction.html) for more insight into specific features.

# How does Bevy work?

Bevy is a low level game engine that uses an ECS architecture to structure it‚Äôs scenes. There are a few concepts you‚Äôll hear me say and use frequently ‚Äî like **entities**, **systems**, **resources**, etc. It‚Äôll become more clear how all these pieces work individually and together as you work with them.

![A diagram of a Bevy app architecture, from plugins like input to resources like Textures to events like NewGameEvent to systems like MovePlayer.](./Bevy_Game_Architecture_R2.png)

You‚Äôll find yourself creating a lot of [‚Äú**systems**‚Äù](https://bevyengine.org/learn/book/getting-started/ecs/#your-first-system) that get added to your Bevy app. Systems are Rust functions that run each frame, usually responsible for logic like moving your character or spawning enemies.

You can use [‚Äú**resources**‚Äù](https://bevyengine.org/learn/book/getting-started/resources/) to manage assets and state in your game. From media like textures and sounds - to data like game state (e.g. ‚Äúis game paused?‚Äù) ‚Äî you‚Äôll use a lot of resources in Bevy. Bevy exposes a bit of it‚Äôs logic through resources, like keyboard and gamepad input (which get added by ‚Äúplugins‚Äù‚Ä¶).

You can organize multiple systems and resources together into a [‚Äú**plugin**‚Äù](https://bevyengine.org/learn/book/getting-started/plugins/). It basically just groups together logic into a single module. It might make sense to make a `GameAudio` plugin that handles all the audio logic in a game. It also makes it easier to enable/disable game features. Bevy provides a few plugins out of the box, like a Window plugin for managing a native window, or an Input plugin for adding a resource to your game with synced user input.

We‚Äôll also end up using [‚Äú**events**‚Äù](https://bevy-cheatbook.github.io/programming/events.html) which are exactly what they sound like. It‚Äôs a way for systems to communicate with other systems using specific events, like a `NewGameEvent` event could play intro music and spawn enemies.

## ‚ÄúMagic‚Äù Function Parameters

The magic of Bevy **systems** is that you can create function parameters for anything you need, and Bevy will automatically provide the system the necessary data.

For example, if you want to access a ‚Äúresource‚Äù, you‚Äôd create a system that has a function parameter with the type `Res<YourResourceHere>`. In this code snippet, we ask for the `Input` and `GameState` resources separately.

```rust
fn move_player(
    keyboard_input: Res<Input<KeyCode>>,
    game_state: Res<GameState>,
) { }
```

You can learn more about this in [the Bevy book](https://bevyengine.org/learn/book/introduction/).

## Creating a ‚Äúplayer‚Äù sprite

Time to create our player! We use `Commands` to spawn a player ‚Äúsprite‚Äù (basically 2D object). This also required making a simple `struct` for the ‚Äú**components‚Äù** for our `Player` and `Collider`. Now lets actually move the player based on the input. In Bevy‚Äôs systems you can `Query<>` for a mutable `Transform` (aka position) of the `Player`. Using that query, we can mutate the value to our new position.

Movement is a simple and clever calculation. We think of movement on a 2D graph using X/Y coordinates, which means moving left would make numbers negative ‚Äî or right positive. Using this as the basis (left is negative, right is positive), we set a `direction` variable with a number that `-1` or `1`, then multiply that by how fast our player should move (`PLAYER_SPEED`). We‚Äôll also multiply by the frame rate of the app (`TIME_STEP`) since the update method runs 60 times a second. Not sure if this is super necessary, cause you can just adjust the other values (like player speed) - but we‚Äôll use this frame rate later to power a timer to keep things more consistent.

```rust
// Defines the amount of time that should elapse between each physics step
// in this case, 60fps
const TIME_STEP: f32 = 1.0 / 60.0;
const PLAYER_SPEED: f32 = 100.0;

fn move_player(
    keyboard_input: Res<Input<KeyCode>>,
    mut query: Query<&mut Transform, With<Player>>,
) {
    let mut paddle_transform = query.single_mut();
    let mut direction = 0.0;

    if keyboard_input.pressed(KeyCode::Left) {
        println!("[KEYBOARD] Pressed left");
        direction -= 1.0;
    }

    if keyboard_input.pressed(KeyCode::Right) {
        println!("[KEYBOARD] Pressed right");
        direction += 1.0;
    }

    // Calculate the new horizontal paddle position based on player input
    let new_paddle_position = paddle_transform.translation.x + direction * PLAYER_SPEED * TIME_STEP;

    paddle_transform.translation.x = new_paddle_position;
}
```

Couple things we need to handle. Projectiles should despawn once they‚Äôre off screen. The top of the screen is around `350`, so we check if the collider‚Äôs translation (aka position) is greater than that. If it is, we ‚Äúdespawn‚Äù the ‚Äúentity‚Äù (aka the projectile and all it‚Äôs components like the `Collider`) to remove the projectile from the game.

```rust
fn destroy_projectiles(
    mut commands: Commands,
    query: Query<(Entity, &Transform), With<Projectile>>,
) {
    for (collider_entity, collider_transform) in &query {
        if collider_transform.translation.y > 350.0 {
            commands.entity(collider_entity).despawn();
        }
    }
}
```

This all works - but if you noticed, we fire like‚Ä¶60 bullets per second üòÖ To prevent this, let‚Äôs create a timer to keep track of the ‚Äúreload‚Äù time of the weapon. This way, the user can only fire again once the timer has completed. It‚Äôll be a short interval (`0.3` seconds).

First let‚Äôs create that timer globally. We create a struct to represent our timer ‚Äúresource‚Äù, and it‚Äôll contain the actual timer as it‚Äôs first tuple property. Then we can insert the resource into our app, and initialize the timer immediately. We have control over the duration of the timer (`0.3` seconds in this case) and if the timer repeats or not. Since we want the user to control the timing of their shot, we‚Äôll do it `Once` ‚Äî so we can reset each time another shot is fired.

```rust

fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .insert_resource(ProjectileTimer(Timer::from_seconds(0.3, TimerMode::Once)))
```

Let‚Äôs update the `shoot_projectiles` function to include a `time` and `projectile_time` parameter. Bevy gives you access to a `Time` resource (I‚Äôm assuming from `CorePlugin`), which is a constantly running timer - so we can keep track of total time elapsed in our game.

Then we have to increment the `ProjectileTimer` timer using the `time.delta()` (or time that passed between last ‚Äúrender‚Äù). **If we don‚Äôt, the timer _will never run_**, and logging out `projectile_timer.elapsed()` will be `0`.

So we have enemies, we have our player shooting projectiles ‚Äî let‚Äôs check if any of the projectiles hit (or ‚Äúcollide) any enemies!

We create a new system called `check_for_collisions` that loops through all the projectiles in a scene, then for each bullet it checks each object in the scene if they‚Äôve collided (using a nice `collide()` method from Bevy‚Äôs 2D system).

Now this trick is nice - since Bevy can query entities for their components, we can also check for components attached to the entity using the `Option<>` enum. In this case, we check if the entity with the `Collider` is also an enemy (`Option<Enemy>`), and if it is, we destroy the enemy by de-spawning it‚Äôs associated entity.

```rust
fn main() {
    App::new()
        .add_plugins(DefaultPlugins)
        .insert_resource(ProjectileTimer(Timer::from_seconds(
            PROJECTILE_TIME_LIMIT,
            TimerMode::Once,
        )))
        .add_startup_system(setup_game)
        .add_system_set(
            SystemSet::new()
                .with_run_criteria(FixedTimestep::step(TIME_STEP as f64))
                .with_system(check_for_collisions)
                .with_system(move_player.before(check_for_collisions))
                .with_system(move_projectiles.before(check_for_collisions))
                .with_system(destroy_projectiles.before(check_for_collisions))
                .with_system(shoot_projectile.before(check_for_collisions)),
        )
        .add_system(bevy::window::close_on_esc)
        .run();
}

fn check_for_collisions(
    mut commands: Commands,
    projectiles_query: Query<(Entity, &Transform), With<Projectile>>,
    collider_query: Query<(Entity, &Transform, Option<&Enemy>), With<Collider>>,
) {
    // Loop through all the projectiles on screen
    for (projectile_entity, projectile_transform) in &projectiles_query {
        // Loop through all collidable elements on the screen
        // TODO: Figure out how to flatten this - 2 for loops no bueno
        for (collider_entity, collider_transform, enemy_check) in &collider_query {
            let collision = collide(
                projectile_transform.translation,
                projectile_transform.scale.truncate(),
                collider_transform.translation,
                collider_transform.scale.truncate(),
            );

            if let Some(collision) = collision {
                // If it's an enemy, destroy!
                if enemy_check.is_some() {
                    println!("Collided!");

                    // Enemy is destroyed
                    commands.entity(collider_entity).despawn();

                    // Projectile disappears too? Prevents "cutting through" a line of enemies all at once
                    commands.entity(projectile_entity).despawn();
                }
            }
        }
    }
}
```

## Backgrounds

The background is looking a bit plain. In Galaga, it‚Äôs a tiled space texture that scrolls to simulate the sense of flying through space.

There‚Äôs a few ways we can do it. If we tried to imitate what we did for the enemies or player ships by creating ‚Äúsprites‚Äù, we‚Äôd have to spawn enough sprites to cover the entire screen ‚Äî then also mutate each position each frame using a system.

To do this more _efficiently_, we‚Äôll implement this logic inside a **fragment shader.** A fragment shader is part of the graphic rendering pipeline and is responsible for ‚Äúpainting‚Äù the pixels down on the screen based on instruction from the previous shaders (like vertex). Since shaders run on the GPU, they‚Äôre better suited for simple (and complex) math operations ‚Äî like iterating over a screen and calculating the color of each pixel.

In Bevy, we can add a ‚Äúshader‚Äù in a few ways. We could apply a shader to [the entire pipeline](https://bevyengine.org/examples/2d/mesh2d-manual/), and change the way the entire scene is rendered. This would be good for post processing, like rendering 100 objects - then blurring all of them at the same time.

Or you could apply a [‚Äúcustom material‚Äù](https://bevyengine.org/examples/shader/shader-material/) to a ‚Äúmesh‚Äù in your scene. This allows us to apply a shader to a specific object, like a cube or 2D shape.

Let‚Äôs create a custom material. You‚Äôll notice we use `Material2d` instead of `Material` because we‚Äôre rendering a 2D scene (not 3D). We also have to give each material a UUID using a macro. And inside the material struct, we add a uniform property for `Color` and a texture (aka optional `Image` `Handle`). Bevy seems to handle generating the sampler for us under the hood, so we only need to pass the image data.

```rust
impl Material2d for CustomMaterial {
    fn fragment_shader() -> ShaderRef {
        "shaders/custom_material.wgsl".into()
    }
}

// Background shader material
#[derive(AsBindGroup, TypeUuid, Debug, Clone)]
#[uuid = "f690fdae-d598-45ab-8225-97e2a3f056e0"]
pub struct CustomMaterial {
    #[uniform(0)]
    color: Color,
    #[texture(1)]
    #[sampler(2)]
    color_texture: Option<Handle<Image>>,
}
```

Create a new file at `/assets/shaders/custom_material.wgsl`. We define our custom material at the top, and grab our uniforms we defined (in this case - only color). Our texture is available through bind groups, and Bevy‚Äôs 2D pipeline puts the texture in the first slot (`@binding(1)`), and sampler in second slot (`@binding(2)`).

> üìò If you try to change these bind group indexes you‚Äôll get an error about bind groups not matching up ‚Äî this is because of how render pipelines work. Bevy defines a specific pipeline configuration for 2D elements, and it assumes the texture will always be bound to slots `1` and `2`. ## Explaining UV Math

What‚Äôs the problem here? Well let‚Äôs break down how UVs work for a second. UVs are usually 2D vectors (aka X and Y values) that go from 0 to 1. The top left of an image is `(0,0)` and the bottom right is `(1,1)`. Our texture data is mapped assuming this, When we multiply the UV by say `4.0`, we‚Äôre telling it to go from `(0,4)` now.

![Diagram of a UV texture that‚Äôs had it‚Äôs coordinates multiplied by 4. A grey square is in center of screen with a 0 in top left corner and 4 in the top right and bottom left. An X axis is scene above the square, and Y to the left. ](./UV_-_Wrong_Example_with_Multiply.png)

Similarly when we divide, we change the range from `(0,1)` to `(0,0.25)` . That‚Äôs what causes the image to look stretched, we‚Äôre only seeing the top left quarter corner of it.

![Diagram of a UV texture that‚Äôs had it‚Äôs coordinates divided by 4. A grey square is in center of screen with a 0 in top left corner and 0.25 in the top right and bottom left. An X axis is scene above the square, and Y to the left. ](./UV_-_Stretching_Example_with_Division.png)

So how do we get our shader to ‚Äúrepeat‚Äù the UV data? The `fract()` function. You can see an [basic interactive example here](https://thebookofshaders.com/09/). By using the `fract()` function, we force the number when it exceeds `1.0` to reset back to `0` and increment up again.

![Diagram of a UV texture that‚Äôs had it‚Äôs coordinates multiplied by 4, then run through a `fract()` function. A grey square is in center of screen. It‚Äôs split into a 4 by 4 grid. Each grid tile has a 0 in top left corner and 1 in the top right and bottom left. An X axis is scene above the grid, and Y to the left. ](./UV_-_Tiling_Example_with_fract.png)

```rust
@fragment
fn fragment(
    #import bevy_pbr::mesh_vertex_output
) -> @location(0) vec4<f32> {
    var tiled_uv: vec2<f32>;
    tiled_uv = fract(uv * 10.0);
    return textureSample(base_color_texture, base_color_sampler, tiled_uv);
}
```

To add a new uniform, go back to our custom material and add a new property under our `color`. We need to use the `uniform()` macro to define the bind group, in this case, we want the same one as our color. We also need to define a data type - in this case `f32`. I originally tried to use a `bool` - but the shader pipeline refused it, so we use a float instead and assume `0` = false and `1` = true.

```rust

    color: Color,
    #[uniform(0)]
    tile: f32,
    #[texture(1)]
    #[sampler(2)]
    color_texture: Option<Handle<Image>>,
}
```

In our shader file we need to update a couple of things. The `CustomMaterial` type at the type needs to have our new `tile` property added to it. Then to access that property, we use the `material` variable (which grabs the data from the bind group we assigned earlier ‚Äî at index `0`). Then we can do a quick conditional check if the tiling is enabled or not.

```rust
struct CustomMaterial {
    color: vec4<f32>,
    tile: f32,
};

@group(1) @binding(0)
var<uniform> material: CustomMaterial;
@group(1) @binding(1)
var base_color_texture: texture_2d<f32>;
@group(1) @binding(2)
var base_color_sampler: sampler;

@fragment
fn fragment(
    #import bevy_pbr::mesh_vertex_output
) -> @location(0) vec4<f32> {
    var tiled_uv: vec2<f32>;
    // Default to the mesh UV
    tiled_uv = uv;
    // If tiling is enabled, redefine the UVs
    if(material.tile > 0.0) {
        var tiled_uv_x: f32;
        var tiled_uv_y: f32;
        tiled_uv_x = fract(uv.x * 10.0);
        tiled_uv_y = fract(uv.y * 7.0);
        tiled_uv = vec2(tiled_uv_x,tiled_uv_y);
    }
    return textureSample(base_color_texture, base_color_sampler, tiled_uv);
}
```

### Animated scrolling background

The background is missing one thing that‚Äôll really sell it - it needs to scroll down like Galaga or most vertical ‚Äúshmups‚Äù (or shoot-em-ups).

To animate anything with shaders, your shaders need to have a sense of time. We‚Äôll create a new uniform for our material called `time`, which will be a `f32` representing time in seconds (e.g `0.0` to infinity). This will let us animated things using math. As the time number changes and gets bigger, we can use it in our shader algorithms to drive movement.

```rust
// Background shader material
#[derive(AsBindGroup, TypeUuid, Debug, Clone)]
#[uuid = "f690fdae-d598-45ab-8225-97e2a3f056e0"]
pub struct CustomMaterial {
    #[uniform(0)]
    color: Color,
    // Should we tile this material? 1 = true
    #[uniform(0)]
    tile: f32,
    // ‚≠ê New uniform for time
    #[uniform(0)]
    time: f32,
    #[texture(1)]
    #[sampler(2)]
    color_texture: Option<Handle<Image>>,
}
```

Let‚Äôs add a new system to our game that goes through all the materials in our scene and updates the time variable on them. When we loop over the materials, they‚Äôre arrays, with the first index being the material ID and the second index as the actual material data. I left a `dbg!()` macro in the function that you can enable to see the time change as a sanity check.

```rust
fn main() {
    App::new()
        .add_system(update_material_time)
}

fn update_material_time(time: Res<Time>, mut materials: ResMut<Assets<CustomMaterial>>) {
    materials.iter_mut().for_each(|material| {
        // dbg!(material.1.time);
        material.1.time = time.elapsed_seconds();
    });
}
```

Now that we have time pumped into our shader, let‚Äôs use it to scroll the background.

Since we want to scroll the screen vertically, we need to focus on the Y coordinate of the UVs. We already take the Y coordinate of the UVs and multiply it, then use `fract()` to make it repeat between `0-1`. We want to stay between the `0-1` coordinates, which means we should put the time inside the `fract()`, so it keeps the ‚Äúrepeating grid‚Äù consistent. If we subtract the time, lets say the UV starts at 0, if we subtract time, the UV will be negative ‚Äî which gets converted back to a positive `0-1` by the `fract()`.

```rust
struct CustomMaterial {
    color: vec4<f32>,
    tile: f32,
    // ‚≠ê Time increments
    time: f32,
};

@group(1) @binding(0)
var<uniform> material: CustomMaterial;
@group(1) @binding(1)
var base_color_texture: texture_2d<f32>;
@group(1) @binding(2)
var base_color_sampler: sampler;

@fragment
fn fragment(
    #import bevy_pbr::mesh_vertex_output
) -> @location(0) vec4<f32> {
    var tiled_uv: vec2<f32>;
    tiled_uv = uv;
    if(material.tile > 0.0) {
        var tiled_uv_x: f32;
        var tiled_uv_y: f32;
        tiled_uv_x = fract(uv.x * 10.0);
        // ‚≠ê We "scroll" the background by subtracting time here
        tiled_uv_y = fract(uv.y * 7.0 - material.time);
        tiled_uv = vec2(tiled_uv_x,tiled_uv_y);
    }
    return textureSample(base_color_texture, base_color_sampler, tiled_uv);
}
```

## Adding sounds

Bevy has a great system setup in place for handling and playing audio assets. Let‚Äôs add some quick sound effects to the game to make it more immersive.

We could play sounds directly in the same functions where we currently fire the projectile - but that function has a lot going on already. Instead lets use Bevy‚Äôs ‚Äúevent system‚Äù to fire off an `EnemyDeathEvent` when we fire the weapon, then we can check for that event inside other systems, like say a `play_enemy_death_sound()` function. We could also create another system/function later that checks for the same event and increases the score.

Let‚Äôs create some structs to represent these ‚Äúevents‚Äù.


We‚Äôve got the event firing off, but we still need to have a system listening for that event and actually playing the sound. So let‚Äôs do just that.

We‚Äôll create a new function `play_projectile_sound` that has an `EventReader` property to read our `ProjectileEvent` events. We also grab a `audio` property that gives us access to Bevy‚Äôs audio system (which we‚Äôll use to play the sound). And we‚Äôll grab the specific sound we want.

We can use the `is_empty()` method on the events to check if the list is empty, and if it‚Äôs not, we first clear the events (so the sound plays only once per frame). Then we can play the sound using the `Audio` system and our sound _(which we have to clone because Rust reasons ü¶Ä)_.

```rust
fn play_projectile_sound(
    projectile_events: EventReader<ProjectileEvent>,
    audio: Res<Audio>,
    sound: Res<ProjectileSound>,
) {
    // Check for events
    if !projectile_events.is_empty() {
        // Clear all events this frame
        projectile_events.clear();
        println!("[AUDIO] Playing projectile sound!");

        audio.play(sound.0.clone());
    }
}
```
